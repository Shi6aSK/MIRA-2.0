1Motivation

Room-scale assistive systems must decide when to wake up, when to sense, and when to interact. This is especially critical in privacy-preserving systems for blind or visually impaired (BVI) users: unnecessary camera or audio activation can expose private data, while missed activation can deny assistance at a critical moment.

Scope. This CPRE 5450 project isolates one dependability-critical subsystem from the larger CPRE 5750 MIRA platform — the activation and presence-verification layer. The central question is:

"When is the system allowed to activate high-power or privacy-sensitive functions such as camera inference, audio capture, or servo actuation?"

This is not only a sensor-fusion problem. It is a fault-tolerant decision problem. The subsystem must remain reliable under sensor faults, timing faults, and environmental uncertainty.

2Problem Statement & Failure Modes

A single-modality activation scheme is brittle — each sensor has different failure conditions. The two major system-level failure modes are:

False Activation

Camera / radar disagrees → system activates anyway

The system activates camera, audio, or actuation even though no valid user is present. Wastes battery and may compromise privacy.

Missed Activation

System fails to activate when needed

The system fails to activate even though the user is present and needs interaction — a denial-of-service failure for an assistive system.

Fault–Error–Failure Chain

Design constraint: The ESP32-S3 has no hardware GPU or neural-network accelerator. All inference runs on the LX7 cores using INT8-quantized models via Espressif's ESP-DL library. Per-frame inference takes 100–200 ms. This motivates a lightweight, rule-based activation controller rather than a heavy deep-learning pipeline.

4Proposed Fault-Tolerant Architecture

Guiding principle: When uncertain, fail into non-activation, not unsafe activation. Ambiguity triggers a safe state, not an active state.

The proposed architecture treats activation as a fault-tolerant state machine. The mmWave sensor acts as a low-power sentinel. When radar indicates possible presence, the camera is activated briefly for confirmation. A fusion layer evaluates agreement, confidence, freshness, and timing health before allowing full activation.

FreeRTOS Task Model

TaskCorePriorityFunction
mmwave_rx_taskCore 0High UART receive, CRC parser, freshness stamp
vision_taskCore 1High Frame capture, format, quality check, inference
ft_taskCore 0MAX-2 Consistency check, scoring, FSM transitions
web_server_taskCore 1Low HTTP dashboard, JPEG stream

The Task Watchdog Timer (TWDT) monitors ft_task for stalls. A stalled task that does not call esp_task_wdt_reset() within the 5-second watchdog period triggers a panic and is logged as a timing fault.

Confidence Score Fusion

Per-sensor confidence is computed each decision window:

Ci = ws · Si + wf · Fi + wt · Ti

  • Si — cross-sensor agreement (1.0 if agree, 0.0 if disagree)
  • Fi — freshness: max(0, 1 − (tnow − tsample) / Tstale)
  • Ti — transport health (1.0 if TWDT not triggered & no queue overflow)

Default weights: w_agreement = 0.50 · w_freshness = 0.30 · w_transport = 0.20

If C < min_confidence (default 0.60), activation is suppressed regardless of binary presence decision.

5Fault Model & Diagnosis Logic

Fault Taxonomy

Faults are classified following Avizienis et al. (2004) along four axes:

By Source

  • Sensing faults — radar clutter, camera low-light, multipath
  • Communication faults — UART drops, PSRAM bus contention, DMA overrun
  • Timing faults — stale frames, over-long inference, watchdog events
  • Environmental faults — fan, curtain, reflective surface, sudden light change

By Duration

  • Transient — last one or a few decision windows; may self-resolve. E.g., brief radar glitch from a passing vehicle.
  • Persistent — persist across k_hard consecutive windows. E.g., camera initialization failure.
  • SDC-like (Silent Data Corruption) — sensor outputs appear valid but conflict with system state.

Fault Detection Rules

Fault ClassExample Observable SymptomResponse
Camera false negative Low light / occlusion Radar present; camera absent Revalidate; soft safe state
Radar false positive Fan / curtain / reflection Radar present; camera no user Delay activation; request confirmation
UART / transport fault Dropped radar packet Missing/stale radar timestamp Parser reset; suppress activation
Timing fault Camera task delay Fusion uses stale frame Reject frame; watchdog recovery
Persistent sensor fault Camera init failure Repeated visual failure Hard safe state; isolate camera
SDC-like output Stale plausible data Valid-looking data conflicts with state Invariant check; rollback health state

6Safe-State & Recovery Policy

Seven-State Activation FSM

IDLE
Low-power sentinel — radar only, camera off.
RADAR_CANDIDATE
Radar has indicated possible presence. Requesting camera.
VISUAL_CONFIRM
Camera confirmation in progress.
ACTIVE
Camera inference, audio capture, and servo actuation allowed.
SOFT_SAFE
Transient inconsistency. Full activation suppressed; retry.
HARD_SAFE
Persistent fault. Activation blocked; notify user.
DEGRADED
One sensor isolated. Minimal low-power operation.

Soft Safe State

Transient Inconsistency

Full activation suppressed. System collects another radar sample and camera frame. If consistency is restored within soft_retry_ms (default 3000 ms), transitions back to Visual Confirm. Otherwise escalates to Hard Safe.

Hard Safe State

Persistent Inconsistency

Automatic activation blocked. Low-confidence sensor input isolated. User optionally notified via OLED/audio. Minimal power operation. Exit requires manual reset or watchdog-supervised sensor re-initialization.

7Evaluation Plan & Metrics

Four Controllers Compared

Baselines
  1. Radar-only — activate on any radar presence signal, no confirmation
  2. Camera-only — activate on any face-detected frame
  3. Naive AND/OR fusion — no fault diagnosis; no safe states
Proposed

FT fusion — confidence scoring, cross-sensor diagnosis, transient/persistent classification, soft/hard safe states, recovery blocks

Fault Injection Tests

Fault TypeInjection MethodTarget Metric
Radar packet drops Drop 1-in-N UART frames randomly FAR, MDR, fault coverage
Radar framing error Corrupt SOF/EOF bytes Fault coverage, detection latency
Camera frame blackout Replace frame buffer with zeros MDR, soft safe state rate
Camera motion blur Convolve with Gaussian kernel Fault coverage
Induced timing delay vTaskDelay injection in sensor task TWDT trigger rate

Key Metrics

MetricDefinition
FAR — False Activation Rate Activations with no valid user / total trials
MDR — Missed Detection Rate No activation with valid user present / total trials
Fault Coverage Injected faults detected and correctly classified / total injected
Detection Latency Time from fault injection to fault flag (ms)
Activation Latency Time from user presence to first allowed activation (ms)
Soft Safe Entry Rate Fraction of trials that enter soft safe state

Expected Results

  • Lower FAR than radar-only — camera provides semantic rejection of clutter.
  • Lower MDR than strict AND fusion — transient faults enter soft safe state and retry.
  • Higher fault coverage than naive fusion — explicit disagreement diagnosis.
  • Moderate latency overhead (<500 ms) from soft-safe revalidation.
